# 🛠️ Gu Gu Gu 系統維護與上線手冊 (終極版)

這份手冊是為了確保你在學校展覽時，能夠快速切換網路、更新內容並解決問題。

---

## 📡 第一部分：更換 WiFi (現場必做)
當你把機器帶到學校時，必須更新 WiFi 帳號密碼：

1. 打開 `esp32cam/esp32cam_push/esp32cam_push.ino`。
2. 修改這兩行：
   ```cpp
   const char* ssid     = "學校WiFi名稱";
   const char* password = "學校WiFi密碼";
   ```
3. 確認 `host` 網址正確：
   ```cpp
   const char* host     = "gugugu-relay.onrender.com";
   ```
4. **燒錄程式** 並按一下 ESP32 上的 Reset 鍵。
5. **確認**：Serial Monitor 看到 `✅ Connected!` 字樣代表影像已經開始上傳雲端。

---

## 💻 第二部分：更新網頁內容
如果你在現場要修改網頁文字、圖片或 CSS，請在電腦上修改後執行：

1. 開啟終端機 (Terminal)。
2. 輸入這三行指令：
   ```bash
   cd C:\Users\astin\Downloads\gugugu
   git add .
   git commit -m "Update website"
   git push
   ```
3. **等它生效**：去 GitHub 的 **Actions** 頁面看，等任務變綠色 ✅ 就代表網頁更新成功了。

---

## ☁️ 第三部分：雲端中繼站 (Render)
中繼站負責把影像發給所有人看，不用每天去動它，但需要注意：

- **喚醒服務**：Render 免費版如果一段時間沒人用會「打瞌睡」。當你早上剛開啟網頁時，可能會看到「連線失敗」，**這時請重新整理網頁**，等待約 30 秒讓雲端伺服器起床即可。
- **連線數**：網頁右側的「連線數」如果跳動，代表伺服器運作正常。

---

## 🖼️ 第四部分：圖片與影片路徑 (重要！)
為了讓 GitHub Pages 讀得到圖片，請遵守：

- **正確路徑**：`./footage/xxx.jpg` (前面有個點)。因為網頁放在子目錄，絕對路徑 `/` 會導致圖裂。
- **大影片**：
  - GitHub 單一檔案上限是 100MB，超過 50MB 會出現警告（但不影響運作）。
  - 如果影片超級大，請不要直接上傳，改放 Google Drive 或 YouTube 並嵌入。
- **Git 檢查**：如果發現某個檔案在電腦上有但網頁上沒出現，請檢查 `.gitignore` 是否不小心忽略了該格式。

## 🆘 緊急檢查清單 (展覽當天)
| 狀況 | 檢查項目 |
| :--- | :--- |
| **沒影像？** | 1. ESP32 電源有沒有插好？ <br> 2. 學校 WiFi 有沒有斷線？ <br> 3. 手動重整一次網頁（Render 喚醒）。 |
| **圖/片出不來？** | 1. 檢查檔名有沒有空白或特殊符號？ <br> 2. 手動執行 `git add [檔名]` 確保檔案有上傳到 GitHub。 |
| **網頁 404？** | 1. 檢查 GitHub Actions 是否變綠色 ✅？ <br> 2. 網址最後有沒有加 `/` (例如 `.../guguguwyhk/`)。 |
| **畫面很卡？** | 1. 學校 WiFi 訊號太弱？ <br> 2. 嘗試重啟 ESP32 或者調低攝像頭解析度。 |

## 📈 第五部分：多人同時使用 (高併發應對)
本專案已針對多人同時在線進行優化：

1. **直播串流 (Relay Server)**：
   - 傳統 ESP32-CAM 只能給 1-2 個人看，但我們使用了 Render 雲端中繼站。它會抓取一個畫面並發送給所有打開網頁的人。
   - **理論上限**：支持 50-100 人同時觀看。如果現場人更多導致卡頓，請稍微調低 ESP32 的 Frame Rate。
2. **投票系統 (Google Sheets Resilience)**：
   - 如果同時有幾十人投票，Google 服務可能會暫時忙碌。我已經加入了 **自動重試機制 (Retry Logic)**。
   - 當連線失敗時，網頁會自動在 1.5 秒後再次嘗試載入榜單，用戶無需手動重新整理。
3. **前端性能**：
   - 網頁已強制開啟 **GPU 硬件加速**，即使多人同時操作互動地圖，畫面也能保持順滑不掉幀。
